The Problem

Your Application Is Sharing a CPU with Interrupt Handlers

In a traditional Linux kernel, application code, TCP stack processing, NIC driver interrupts, and NVMe completions all compete for the same cores. Softirqs steal cycles from your application under load. The Linux community has spent years building workarounds: isolcpus, irqaffinity, RPS/RFS, SO_BUSY_POLL, NAPI threaded mode. All are imperfect because within one kernel, bottom-half work can always sneak onto app cores.

Split-Kernel Architecture

Many App-Kernels, One Device-Kernel

Each application gets its own app-kernel with dedicated cores and zero interrupts. A shared device-kernel handles all drivers, interrupts, and network stack on separate cores. Connected through shared memory.

Why This Works

Linux already treats application code and interrupt processing as separate contexts. We make that separation physical, giving each application its own kernel on its own cores.

Predictable Latency

No interrupt can fire on an app core. No softirq can steal cycles. Your application gets deterministic CPU time without any kernel tuning.

Fault Isolation

A NIC driver crash takes down the device-kernel, not your application. The app-kernel keeps running while you recover or replace the device-kernel.

No Configuration Required

No isolcpus, no irqaffinity, no RPS/RFS tuning. The bottom half cannot run on app cores because it is in a different kernel.

Docker Compatible

Boot directly into Docker images. DAXFS shares the container rootfs across kernel instances. Use your existing images and workflow.

Works in Cloud VMs

No hypervisor needed. No nested virtualization. Multikernel runs on bare metal or inside any cloud VM on AWS, GCP, or Azure.

100% Open Source

Built on upstream Linux. All kernel patches, DAXFS, and management tools are open source under GPL-2.0. No proprietary components.
