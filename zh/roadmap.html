Multikernel 系统设计与路线图

我们通过 multikernel 架构革新云操作系统的完整规划。

摘要： Multikernel 项目引入了一种全新的 Linux 操作系统架构，可在不依赖虚拟化的情况下运行多个 Linux 内核实例，每个核心或核心组独立运行专用的内核实例。这种方式解决了传统单内核与微内核设计在可扩展性上的局限，同时提供更强的性能隔离、弹性资源分配以及零停机升级能力。

1. 设计理念

1.1 灵活性优先

该设计旨在通过可编程接口最大化灵活性，广泛利用 eBPF 实现动态行为修改，无需重新编译内核。

1.2 选择的自由

该设计必须保留并尊重用户的选择权，包括启用与禁用此功能，以及与所有合理的现有竞争方案配合使用，例如在其之上使用 SR-IOV 或通用虚拟化。

1.3 简洁与极简

该设计通过避免困扰当前虚拟化技术栈的复杂抽象层，保持架构上的简洁性。

1.4 基础设施复用

该设计应尽可能复用现有内核子系统，包括用于内核加载的 kexec、用于资源管理的 CPU/内存热插拔、用于 I/O 的现有驱动框架，以及用于可编程性的标准 eBPF 基础设施。

2. 目标应用场景

2.1 高性能工作负载隔离

提供一种具备更优性能特性的容器和虚拟机替代方案。每个应用程序获得具有定制化配置的专用内核实例，消除嘈杂邻居效应，同时保持接近裸金属的性能。弹性资源分配能够根据工作负载需求动态扩缩容，而无需承受传统虚拟化的开销。

2.2 内核定制与专用化

使用户能够通过多种机制部署面向应用的定制内核配置：用于运行时行为修改的 eBPF 程序、用于硬件优化的专用内核模块，以及用于工作负载适配的机器学习驱动参数调优。

这适用于从需要微秒级延迟的高频交易，到需要专用内存管理的科学计算等各类场景。

2.3 内核级故障容错

实现故障隔离，使内核实例能够独立发生故障，而不影响其他实例或宿主系统。发生故障的实例可以在保持应用程序可用性的同时被透明地重启或替换。与内核故障会影响整个系统的单内核架构（例如用于虚拟机的虚拟机管理程序内核）相比，这大幅提升了可靠性。

2.4 零停机内核热升级

通过生成带有更新版本的新内核实例，并逐步将工作负载从旧实例迁移过去，支持无缝的内核更新。这使系统能够在安全补丁、功能更新或配置变更期间持续运行，满足生产环境中对正常运行时间的关键要求。

3. 实施路线图

3.1 内核加载基础设施增强

当前状态：已使用 kexec 实现基本的内核加载。
目标：
  • 迁移到基于 C 的跳板（trampoline）实现，以提升可维护性与可移植性
  • 实现基于 kexec --unload 的关闭机制，以实现内核实例的干净终止
  • 通过 kexec_file_load() 增加内核镜像验证与安全启动集成支持

3.2 内核间通信基础设施

当前状态：已建立基本的 IPI（处理器间中断）通信，并具备初步的共享内存支持。
目标：
  • 基于 IPI 与共享内存原语，开发全面且灵活的消息传递协议
  • 为内核间通信建立安全边界与访问控制
  • 该基础设施将作为资源管理与升级协议的基础

3.3 动态硬件资源管理

当前状态：尚未实现；依赖通信基础设施的完成。
目标：
  • CPU 管理：与 CPU 热插拔子系统集成，实现动态核心分配与迁移
  • 内存管理：利用 CMA（连续内存分配器）与内存热插拔实现弹性内存分配
  • 中断投递：实现高性能门铃（doorbell）机制，以高效处理硬件中断
  • I/O 资源分配：使用硬件队列管理取代 SR-IOV，实现细粒度的 I/O 资源分区
    • 网络 I/O
    • 存储 I/O
    • GPU
  • eBPF 集成：通过 eBPF 程序启用可编程的资源分配策略，实现自适应资源管理

3.4 零停机内核热升级实现

当前状态：尚未实现；需要前述全部基础设施组件完成后方可进行。
目标：
  • 在内核交接（Kernel Hand Over，KHO）之上设计并实现协议，用于内核实例间的协同状态传递
  • 开发状态迁移机制，在内核过渡期间保留应用程序上下文
  • 创建编排逻辑，用于管理跨多个内核实例的升级序列
  • 为升级失败场景实现回滚能力
  • 与现有更新机制及包管理系统（如 NixOS）集成

3.5 与 Kubernetes 集成

当前状态：尚未实现；需要前述全部基础设施组件完成后方可进行。
目标：
  • 开发完善的 Kubernetes 容器运行时接口（CRI）插件，实现对 multikernel 实例的无缝编排
  • 为 multikernel 专属配置与策略实现自定义资源定义（CRD）
  • 创建 multikernel 调度器扩展，根据内核实例能力与资源需求优化 Pod 调度
  • 设计与 Kubernetes 网络（CNI）及存储（CSI）接口在 multikernel 环境中的集成方案
  • 建立与 Kubernetes 指标及日志系统的监控与可观测性集成
  • 在 Kubernetes 工作流中，为内核实例的创建、扩缩容与终止实现生命周期管理钩子

结语

Multikernel 项目代表着向分布式内核架构的根本性转变，旨在解决当前系统在可扩展性与灵活性方面的局限。项目的成功取决于对五阶段路线图的审慎实施，每个阶段都将为后续组件构建必要的能力。对灵活性、选择权、简洁性与基础设施复用的重视，确保最终系统在与现有云计算生态系统保持兼容的同时，提供切实的实际收益。
