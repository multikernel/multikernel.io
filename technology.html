Technology Overview

Multiple Linux Kernels, One Machine

Run independent Linux kernel instances simultaneously on bare metal. Each kernel gets dedicated CPUs, memory, and devices with native performance and no hypervisor overhead.

Core Concept

Kernel-Level Isolation Without Virtualization

Traditional approaches force a choice: containers share a single kernel (weak isolation), or VMs add a hypervisor layer (performance overhead). Multikernel eliminates this tradeoff by running multiple Linux kernels directly on hardware.

Separate Kernels, Not Shared

Each workload runs in its own kernel instance. A kernel vulnerability in one instance cannot affect the others.

Native Hardware Access

No hypervisor, no emulation layer. Kernels access CPUs, memory, NICs, and GPUs directly with zero virtualization overhead.

Per-Workload Kernel Configuration

Each kernel instance can be configured and tuned independently. A database gets an I/O-optimized kernel; a web server gets a network-optimized one.

Under the Hood

How It Works

Built entirely on existing Linux infrastructure. No new OS, no custom hypervisor, no compatibility gaps.

Kernel Spawning via Kexec

New kernel instances are launched using Linux's kexec mechanism, extended to boot alongside the running kernel rather than replacing it. The spawned kernel starts on its assigned CPUs and memory, fully independent of the primary kernel.

Dynamic Resource Partitioning

CPUs, memory, and devices are assigned to kernel instances using Linux's standard hotplug interfaces. Resources can be rebalanced between kernels at runtime without rebooting or restarting applications.

Hardware Queue Sharing

Modern NICs and SSDs expose multiple hardware queues. Each kernel instance gets exclusive access to specific queues, providing true hardware-level I/O isolation without SR-IOV or software bridges.

Inter-Kernel Communication

Kernel instances communicate over shared memory using vsock, a standard Linux socket type. Applications use familiar socket APIs with no code changes required.

Multikernel vs. Alternatives

Multikernel combines the isolation of VMs with the performance of bare metal and the flexibility of per-workload kernel customization.

Applications

Use Cases

Multikernel is designed for workloads that need strong isolation, dedicated resources, and native hardware access.

Workload Isolation

Run mixed workloads on the same machine with kernel-level isolation. Each workload gets its own kernel, scheduler, and resources. No noisy neighbor effects.

Zero-Downtime Kernel Updates

Boot a new kernel alongside the running one, migrate workloads over, then retire the old kernel. Roll back at any point if something goes wrong.

AI Agent Sandboxing

Give each AI agent its own kernel with full GPU access and strong isolation. No virtualization layer between the agent and hardware. Works inside cloud VMs.

Kernel Fault Tolerance

Run a standby kernel that monitors the primary via heartbeat. If the primary crashes, the standby takes over its devices and resumes service in sub-second time.

Docker Integration

Boot directly into Docker images with DAXFS sharing the container rootfs across kernel instances. No full OS per instance, just the kernel and your application.

CXL Memory Sharing

Share data across kernel instances or CXL-connected hosts through DAXFS, with lock-free concurrent access and a shared page cache in DAX memory.
