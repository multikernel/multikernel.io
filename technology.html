Technology Overview

Split-Kernel Architecture

Separate application logic from device processing into independent Linux kernels. The app-kernel runs your code with zero interrupts. The device-kernel handles all I/O. Connected through shared memory.

Get Started View Source
The Problem

Interrupts Are Stealing Your Application's CPU

In a traditional Linux kernel, your application shares cores with the TCP stack, NIC drivers, NVMe completions, and every softirq handler. Under load, interrupt processing steals cycles from your application. The kernel community has spent years building workarounds.

isolcpus and irqaffinity

Pin interrupts to specific cores. But softirqs can still sneak onto app cores under load, and configuration is fragile.

NAPI threaded mode and SO_BUSY_POLL

Move poll processing to threads or busy-wait in the application. Trading CPU for latency, still within the same kernel.

RPS/RFS and XPS

Steer packets to specific cores. Helps with locality but cannot prevent interrupt coalescing or softirq bursts on the wrong core.

The Solution

Split the Kernel at the Interrupt Boundary

Instead of tuning one kernel to keep interrupts away from your app, split them apart. Each application gets its own app-kernel with zero interrupts. A shared device-kernel handles all drivers and I/O processing on dedicated cores.

App-Kernels

Each application gets its own kernel running syscalls and VFS on dedicated cores. Zero interrupts, ever. Run as many app-kernels as your workloads need.

Device-Kernel

A shared kernel that runs the TCP stack, NIC driver, NVMe block layer, and all interrupt handlers. All I/O processing is contained here, serving every app-kernel through shared memory.

Under the Hood

How It Works

Built entirely on existing Linux infrastructure. No new OS, no custom hypervisor, no compatibility gaps.

Kernel Spawning via Kexec

New kernel instances are launched using Linux's kexec mechanism, extended to boot alongside the running kernel rather than replacing it. The spawned kernel starts on its assigned CPUs and memory, fully independent of the primary kernel.

Dynamic Resource Partitioning

CPUs, memory, and devices are assigned to kernel instances using Linux's standard hotplug interfaces. Resources can be rebalanced between kernels at runtime without rebooting or restarting applications.

Hardware Queue Sharing

Modern NICs and SSDs expose multiple hardware queues. Each kernel instance gets exclusive access to specific queues, providing true hardware-level I/O isolation without SR-IOV or software bridges.

DAXFS Shared Memory

The app-kernel and device-kernel communicate through DAXFS, a filesystem built on shared DRAM with lock-free coordination. Zero-copy, no serialization, no network round trips.

Applications

Use Cases

Split-kernel architecture is designed for workloads where interrupt jitter, I/O isolation, or independent kernel lifecycle matters.

Latency-Sensitive Workloads

Trading systems, real-time analytics, and game servers where interrupt jitter is unacceptable. App cores never see an interrupt.

Zero-Downtime Kernel Updates

Boot a new kernel alongside the running one, migrate workloads over, then retire the old kernel. Roll back at any point if something goes wrong.

AI Agent Sandboxing

Give each AI agent its own kernel with full GPU access and strong isolation. No virtualization layer between the agent and hardware. Works inside cloud VMs.

Kernel Fault Tolerance

Run a standby kernel that monitors the primary via heartbeat. If the primary crashes, the standby takes over its devices and resumes service in sub-second time.

Docker Integration

Boot directly into Docker images with DAXFS sharing the container rootfs across kernel instances. No full OS per instance, just the kernel and your application.

CXL Memory Sharing

Share data across kernel instances or CXL-connected hosts through DAXFS, with lock-free concurrent access and a shared page cache in DAX memory.
